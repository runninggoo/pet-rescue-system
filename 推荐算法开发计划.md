## 一、功能目标

在宠物列表页顶部，为已登录用户展示"为你推荐"模块，基于用户历史行为（浏览/收藏/申请）计算偏好，个性化推荐宠物列表。

---

## 二、算法流程（文字描述）

### 第一步：行为采集

当用户执行以下操作时，向 `user_pet_behavior` 表写入一条行为记录：

| 行为 | 类型标识 | 权重分 |
|------|---------|--------|
| 浏览宠物详情 | `view` | +1 |
| 收藏宠物 | `favorite` | +3 |
| 申请领养 | `apply` | +5 |

采集方式：宠物详情页加载时，前端静默调用新增的 `/api/recommendation/record` 接口，**不影响原有任何业务逻辑**。

### 第二步：计算用户偏好向量

```sql
SELECT p.category_id,
       SUM(CASE pb.behavior_type
             WHEN 'apply'   THEN 5
             WHEN 'favorite' THEN 3
             WHEN 'view'    THEN 1
           END) AS score
FROM user_pet_behavior pb
JOIN pet p ON pb.pet_id = p.id
WHERE pb.user_id = #{userId}
GROUP BY p.category_id
ORDER BY score DESC
LIMIT 3
```

将得分最高的 3 个分类作为用户的核心偏好（归一化为权重）。

### 第三步：推荐宠物打分

对用户**未产生过行为的宠物**进行打分：

```
Score = α × 类别匹配度 + β × 紧急程度 + γ × 新鲜度
```

| 维度 | 计算方式 | 权重 |
|------|---------|------|
| 类别匹配度 | 核心偏好分类 → 1.0，第二偏好 → 0.5，其余 → 0.0 | α = 0.5 |
| 紧急程度 | 紧急 → 1.0，一般 → 0.5 | β = 0.3 |
| 新鲜度 | 7天内 → 1.0，7-30天 → 0.6，30天+ → 0.2 | γ = 0.2 |

### 第四步：返回TopN推荐

取得分最高的 6 条宠物详情返回给前端。

**兜底逻辑**：用户行为少于3条时，切换为热门宠物兜底（status=1 按 created_at DESC limit 6）。

## 五、接口设计（独立API）

### 接口一：行为记录

```
POST /api/recommendation/record
  入参：{ petId: Long, behaviorType: String }  // 'view'|'favorite'|'apply'
  出参：{ code: 200, msg: 'ok' }
  说明：静默记录行为，不影响原有业务逻辑
```

### 接口二：个性化推荐

```
GET  /api/recommendation/personalized
  参数：limit（默认6）
  出参：{ code: 200, data: [ PetVO, ... ] }
  说明：仅登录用户可用，未登录返回空数组
```

**不影响任何原有接口**，原有 `/api/pet/list`、`/api/auth/login` 等全部保持原样。

## 九、防Bug保障措施

### 1. 完全隔离原则
- 新增文件全部使用新包名：`UserPetBehavior*`、`Recommendation*`、`BehaviorRecord*`
- **零修改**任何现有 Controller/Service/Mapper
- 数据库**零修改**任何现有表结构

### 2. 空值保护
- 用户无行为记录 → 返回空数组，前端 `v-if` 兜底
- 推荐结果不足6条 → 返回实际数量，前端正常渲染
- 宠物分类为 null → 跳过，不参与打分

### 3. 性能保护
- 推荐查询设置超时（3秒），超时时降级返回热门宠物
- 行为记录写入**异步化**（`@Async`），不阻塞原有业务

### 4. 幂等性
- `user_pet_behavior` 表设置联合唯一键 `(user_id, pet_id, behavior_type)`
- 同一用户对同一宠物重复"浏览"行为 → 自动忽略，不重复写入

### 5. 回归测试
- 开发完成后，用管理员/领养人/志愿者账号逐一测试原有16个页面
- 特别验证：pet-list.html 原有搜索/筛选/分页不受任何影响
