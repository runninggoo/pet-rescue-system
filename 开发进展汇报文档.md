## 一、项目概述

### 1.1 项目背景

上海地区流浪宠物救助收养领域长期面临信息不对称、资源分散、流程不规范等问题。本系统旨在通过数字化手段覆盖"救助发现 → 健康管理 → 领养审核 → 后续跟踪"全业务流程，提升救助效率，降低流浪宠物死亡率。

### 1.2 技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 后端框架 | Spring Boot 2.7.0 | 主流 Java 框架，稳定可靠 |
| 安全认证 | Spring Security + JWT | 无状态认证，支持多角色 |
| 持久层 | MyBatis Plus 3.5.3.1 | 简化 CRUD，支持逻辑删除 |
| 前端框架 | Vue.js 2.6.14 + Element UI 2.15.13 | 组件化开发，交互友好 |
| 构建工具 | Vite（开发服务器） | 快速热更新 |
| 数据库 | MySQL 8.0 | 结构化数据存储 |
| 图表库 | ECharts | 数据可视化统计 |

### 1.3 系统角色（5种）

| 角色 | 角色标识 | 主要权限 |
|------|---------|---------|
| 管理员 | ADMIN | 全局管理、宠物增删、审核申请、数据统计 |
| 机构管理员 | INSTITUTION_ADMIN | 本机构宠物管理、申请审核 |
| 领养人 | ADOPTER | 浏览宠物、提交领养申请、查看进度 |
| 志愿者 | VOLUNTEER | 领取任务、完成任务、积分成长 |
| 宠物医院 | HOSPITAL | 查看关联宠物健康档案 |

## 二、系统功能模块

### 2.1 模块一：用户认证与权限系统

**功能说明**：支持 5 种角色的注册/登录/JWT 认证，RBAC 权限控制。

**核心文件**：

```
backend/src/main/java/com/pet/rescue/
├── controller/AuthController.java          # 登录/注册接口
├── security/JwtAuthenticationFilter.java    # JWT 过滤器
├── security/UserContext.java                # 当前用户上下文
└── config/SecurityConfig.java              # 安全配置
```

**实现要点**：

1. **JWT 无状态认证**：登录成功后服务端生成 Token，客户端每次请求携带在 Header 中，过期需重新登录
2. **Spring Security 过滤链**：JwtAuthenticationFilter 拦截所有请求，验证 Token 后设置 SecurityContext
3. **RBAC 权限控制**：Controller 层使用 `@PreAuthorize` 注解控制接口权限，数据层通过 UserContext 过滤
4. **隐私脱敏**：手机号展示为 `138****0000`，姓名展示为 `张*`，保护用户隐私

**API 接口**：

```
POST /api/auth/login    — 手机号+密码登录，返回 JWT Token
POST /api/auth/register — 用户注册（手机号+密码+角色）
GET  /api/auth/info    — 获取当前用户信息
POST /api/auth/logout   — 退出登录
```

### 2.3 模块三：救助站智能推荐（核心亮点）

**功能说明**：用户捡到流浪宠物后，系统自动推荐附近有剩余容量的救助站，解决"不知道送哪里去"的实际痛点。

**算法原理**：

```
1. 获取用户位置（GPS 或手动输入地址/经纬度）
2. 查询所有已入驻且状态正常的救助站
3. 使用 Haversine 公式计算每个救助站与用户的距离
   d = 2R × arcsin(√(sin²((lat2-lat1)/2) + cos(lat1)×cos(lat2)×sin²((lon2-lon1)/2)))
4. 筛选距离 ≤ maxDistance（默认 10 公里）的救助站
5. 计算综合评分：
   综合评分 = 距离评分×40% + 容量评分×30% + 医疗等级评分×30%
   - 距离评分 = 1.0 / (1.0 + distance/10)
   - 容量评分 = 1.0 - (currentCapacity / maxCapacity)
   - 医疗等级评分 = medicalLevel × 0.2
6. 按综合评分降序返回推荐结果
```

**容量状态判定**：

| 状态 | 条件 | 标识 |
|------|------|------|
| 充足 | 剩余容量 ≥ 30% | 🟢 |
| 紧张 | 剩余容量 10%~30% | 🟡 |
| 已满 | 剩余容量 < 10% | 🔴 |

**核心文件**：

```
backend/src/main/java/com/pet/rescue/
├── controller/ShelterRecommendationController.java
├── service/impl/ShelterServiceImpl.java
├── dto/ShelterRecommendationRequest.java
└── vo/ShelterRecommendationVO.java

frontend/public/shelter-recommendation.html
```

**API 接口**：

```
POST /api/shelter-recommendation/recommend
Body: { "latitude": 31.2397, "longitude": 121.4998, "maxDistance": 10 }
Response: [{ shelterId, name, distance, score, capacityStatus, mapUrl }, ...]
```

### 2.4 模块四：领养申请管理

**功能说明**：领养人提交申请，机构管理员初审，管理员终审，线上签署领养协议。

**申请状态机（6种状态）**：

```
待审核(0) → 已通过(1) → 待签署(4) → 已完成(5)
     ↓          ↓
  已拒绝(2)  待补充(3)
```

**实现要点**：

1. **竞争性申请**：多个领养人可同时申请同一宠物，管理员择优审核
2. **领养协议**：审核通过后进入签署流程，领养人在线确认协议
3. **审核历史**：每次审核记录含审核意见和时间，可追溯

**核心文件**：

```
backend/src/main/java/com/pet/rescue/
├── controller/AdoptionApplicationController.java
├── service/impl/AdoptionApplicationServiceImpl.java
└── entity/AdoptionApplication.java

frontend/public/adoption-list.html
```

**API 接口**：

```
POST /api/adoption/apply              — 提交申请（领养人）
GET  /api/adoption/my-applications    — 我的申请（领养人）
```

## 三、已解决的关键技术问题

### 3.1 志愿者任务统计 Bug（2026-04-04）

**问题**：`getStats(userId)` 方法接收 userId 但从未使用，四个 selectCount 均为全系统统计，导致个人中心统计与任务列表数据完全矛盾。

**修复**：所有计数条件统一加 `.eq(VolunteerTask::getVolunteerId, userId)`。

### 3.2 品种分类 Bug（2026-04-03）

**问题**：`getChildren(parentId)` 误用 `isNull()` → 查出根节点而非子节点，所有大类品种混在一起。

**修复**：`isNull` → `isNotNull`；批量查询改为一次性查出所有子节点并按 parentId 分组。

### 3.3 MyBatis Plus 逻辑删除 Bug（2026-04-02）

**问题**：`application.yml` 全局配置 + `@TableLogic` 注解双重生效，加上数据库 deleted=NULL，导致 COUNT=0。

**修复**：移除 application.yml 全局配置，仅使用 `@TableLogic` 注解；手动处理 deleted=NULL 的数据。

### 3.4 Axios baseURL 导致全站 404（2026-04-01）

**问题**：Axios 1.x 中 `defaults.baseURL` 与相对路径有 bug，路径被重复拼接。

**修复**：删除所有 baseURL 配置，所有 axios 调用改用完整路径 `/api/...`。
