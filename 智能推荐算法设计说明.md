# 救助站智能推荐算法设计

## 一、算法概述

本系统采用基于权重评分的智能推荐算法，综合考虑距离、容量和医疗等级三个维度，为用户推荐最优的救助站。

**推荐策略**：桌面端优先使用 IP 定位（自动感知用户大致位置），移动端优先使用 GPS 精确定位，两种方案均作为兜底保障用户始终能获取推荐结果。

---

## 二、评分维度与权重

| 维度 | 权重 | 说明 |
|------|------|------|
| 距离 | 40% | 用户到救助站的距离，越近越优先 |
| 容量 | 30% | 救助站剩余容量，越多越优先 |
| 医疗等级 | 30% | 救助站医疗设施等级，越高越优先 |

---

## 三、评分计算公式

### 1. 综合评分

```
总分 = 距离得分 × 0.4 + 容量得分 × 0.3 + 医疗等级得分 × 0.3
```

### 2. 距离得分计算（指数衰减）

```java
距离得分 = 100 × e^(-3 × distance / maxDistance)
```

- `distance`: 用户到救助站的直线距离（公里）
- `maxDistance`: 用户设置的最大搜索范围（公里）
- 使用指数衰减函数，距离越远得分越低，且衰减速度适中

**示例：**
- 距离为0km时，得分 = 100分
- 距离为maxDistance/3时，得分 ≈ 36.8分
- 距离为maxDistance时，得分 ≈ 5分

### 3. 容量得分计算

```java
容量得分 = 50 + (剩余容量 / 最大容量) × 50
```

- 基础分50分（保证即使容量很少也有基本分数）
- 比例加分最多50分（剩余容量占最大容量的比例 × 50）

**示例：**
- 剩余容量为0时，得分 = 50分
- 剩余容量为50%时，得分 = 75分
- 剩余容量为100%时，得分 = 100分

### 4. 医疗等级得分

| 医疗等级 | 描述 | 得分 |
|----------|------|------|
| 0 | 无医疗服务 | 25分 |
| 1 | 基础医疗服务 | 50分 |
| 2 | 专业医疗服务 | 75分 |
| 3 | 综合医疗服务 | 100分 |

```java
医疗等级得分 = (医疗等级 + 1) × 25
```

## 五、排序策略

1. **主要排序**：按综合评分降序排列（分数越高越靠前）
2. **次要排序**：当综合评分相同时，按距离升序排列（距离越近越靠前）

## 六、算法伪代码

```
输入: userLocation(纬度, 经度), maxDistance, limit
输出: 推荐的救助站列表

1. 查询所有符合条件的救助站:
   - 状态正常 (status = 0)
   - 已入驻 (entryStatus = 1)
   - 已审核通过 (auditStatus = 1)
   - 有剩余容量 (如果 needCapacity = true)

2. 对每个救助站计算:
   a. distance = 计算Haversine距离(userLocation, shelterLocation)
   b. IF distance > maxDistance: 跳过
   c. distanceScore = 100 × exp(-3 × distance / maxDistance)
   d. capacityScore = 50 + (remainingCapacity / maxCapacity) × 50
   e. medicalScore = (medicalLevel + 1) × 25
   f. totalScore = distanceScore × 0.4 + capacityScore × 0.3 + medicalScore × 0.3

3. 按 totalScore 降序排序

4. 返回前 limit 个结果
```

## 七、位置获取方案（网页端核心设计）

### 7.1 方案选型背景

网页端（而非原生 App）获取用户位置面临以下挑战：

| 场景 | 问题 | 解决方案 |
|------|------|---------|
| 桌面浏览器 | 用户可能拒绝 GPS 定位授权，或浏览器不支持 Geolocation API | **IP 定位兜底**，自动感知大致位置 |
| 移动端浏览器 | GPS 信号弱或室内环境定位不准 | **IP 定位作为补充**，GPS 优先 |
| 所有浏览器首次访问 | 用户未授权时无法获取位置 | **预设多测试位置**，手动选择演示 |
| 演示场景 | 答辩/测试时需要模拟不同区域 | **10 个预置测试位置**，覆盖上海各区 |

### 7.3 各方案技术细节

#### GPS 定位（移动端首选）

```javascript
// 使用浏览器原生 Geolocation API
navigator.geolocation.getCurrentPosition(
    success => {
        // position.coords.latitude: 纬度
        // position.coords.longitude: 经度
        // position.coords.accuracy: 精度（米）
    },
    error => {
        // error.code: 1=拒绝, 2=无法获取, 3=超时
    },
    {
        enableHighAccuracy: true,  // 高精度模式
        timeout: 10000,            // 超时10秒
        maximumAge: 60000         // 缓存60秒
    }
);
```

**特点**：精度高（误差5-50m），但需要用户授权，室内/信号差时不稳定。

#### IP 定位（桌面端首选 / GPS 兜底）

```javascript
// 调用免费 IP 定位服务（ipapi.co）
// 无需 API Key，免费版 45 次/分钟，演示场景足够
fetch('https://ipapi.co/json/?fields=latitude,longitude,city,country_name')
    .then(res => res.json())
    .then(data => {
        // data.latitude: 纬度
        // data.longitude: 经度
        // data.city: 城市名称
    });
```

**特点**：无需用户授权，自动获取，精度中等（误差500m-5km，适合"附近救助站"场景）。

## 九、算法优势

### 1. 多维度综合评估
不像简单的按距离排序，本算法综合考虑三个对用户决策重要的因素。

### 2. 合理的权重分配
- 距离权重最高(40%)：用户最关心是否能快速送达
- 容量和医疗各占30%：同等重要，用户可根据实际情况选择

### 3. 非线性距离衰减
使用指数衰减函数，远距离的惩罚更明显，符合用户心理预期。

### 4. 有界得分范围
所有维度的得分都在0-100之间，便于横向比较和理解。

### 5. 位置获取双模式保障
- 移动端 GPS 优先，确保精度
- 桌面端 IP 定位兜底，无需授权
- 演示场景预置测试位置，覆盖上海各区

## 十、性能优化

1. **索引优化**：在数据库的 lat、lon 字段上建立空间索引
2. **预计算**：提前计算固定路线距离
3. **缓存**：热门区域的推荐结果缓存
4. **分页**：限制返回数量，避免大数据量传输

## 附录A：Haversine 公式（距离计算原理）

地球是一个近似球体，半径为 6371km。两点间的大圆距离通过 Haversine 公式计算：

```java
// Haversine 公式实现
a = sin²(Δlat/2) + cos(lat1) × cos(lat2) × sin²(Δlon/2)
c = 2 × atan2(√a, √(1-a))
distance = R × c  // R = 6371km
```
