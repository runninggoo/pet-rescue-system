### 4.3 P1 - 品种多级分类管理（简化版）

**现状**：宠物品种字段目前是简单的字符串（如"橘猫"），没有层级结构。

**实现方案（推荐：邻接表模型，1天，简化版不做三级分类）**：

**数据库设计**：

```sql
-- 新增 pet_category 表（品种分类表）
CREATE TABLE pet_category (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL COMMENT '分类名称，如"猫"、"橘猫"',
    parent_id BIGINT DEFAULT NULL COMMENT '父级ID，NULL表示一级分类',
    type VARCHAR(20) NOT NULL COMMENT '大类：CATS/DOGS/SMALL_ANIMALS/BIRDS/OTHER',
    sort INT DEFAULT 0 COMMENT '排序序号',
    is_active TINYINT DEFAULT 1 COMMENT '是否启用：0-禁用 1-启用',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (parent_id) REFERENCES pet_category(id) ON DELETE SET NULL
);

-- 初始数据（一级分类）
INSERT INTO pet_category (name, parent_id, type, sort) VALUES

('鸟类', NULL, 'BIRDS', 4),

-- 二级分类示例（橘猫=猫的子分类）
INSERT INTO pet_category (name, parent_id, type, sort) VALUES

```

**API设计**：

```java
// 1. 获取一级分类（按type过滤）
GET /api/category/list?type=CATS
// 返回：[{"id":1,"name":"猫","children":[]}]

// 2. 获取子分类
GET /api/category/children/{parentId}
// 返回：[{"id":6,"name":"橘猫"},{"id":7,"name":"波斯猫"}]

// 3. 管理员：添加分类
POST /api/category/add
// { "name": "布偶猫", "parentId": 1, "type": "CATS" }

// 4. 管理员：删除分类
DELETE /api/category/delete/{id}
// （有子分类或关联宠物时不允许删除）

// 5. 管理员：修改分类
PUT /api/category/update/{id}
// { "name": "新名称", "sort": 1 }
```

### 4.5 P1 - 数据安全功能（替代实名制的新方案）

**取消实名制的理由**（重要）：

> 实名制的三大风险（答辩致命伤）：
> 1. **无真实核验接口**：你无法证明身份证是真实的，评委一句话就能问死
> 2. **隐私数据法律风险**：存储身份证号若无资质涉嫌违反《个人信息保护法》
> 3. **人工审核无法演示**：没有真实用户申请，审核功能形同虚设

**数据安全方案（三合一，成本低，论文价值更高）**：

#### 子功能1：密码BCrypt加密完善（已有基础，强化即可）

当前系统已有 BCrypt 加密，可补充：
- 密码强度校验（前端+后端双重校验）
- 登录失败次数限制（防止暴力破解）
- 论文可写："用户密码采用BCrypt单向哈希加盐存储，即使数据库泄露也无法还原明文密码"

#### 子功能2：隐私信息脱敏展示（3行代码，立刻可做）

| 字段 | 明文存储 | 脱敏展示 |
|------|---------|---------|
| 手机号 | 13800000000 | 138\*\*\*\*0000 |
| 真实姓名 | 张三 | 张\* |

```javascript
// 手机号脱敏（前端一行）
function maskPhone(phone) {
    return phone.replace(/(\d{3})\d{4}(\d{4})/, '$1****$2');
}
```

#### 子功能3：操作日志记录（中等成本，论文价值高）

记录用户在系统中的关键操作（登录、提交申请、审核等），用于：
- 数据安全审计追踪
- 论文第五章"系统安全测试"有内容可写
- 毕设答辩时回答"你的系统怎么保证数据安全"有实质内容

```sql
CREATE TABLE operation_log (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    username VARCHAR(50),
    operation VARCHAR(100) COMMENT '操作类型：LOGIN/LOGOUT/APPLICATION/AUDIT...',
    detail TEXT COMMENT '操作详情JSON',
```

### 4.6 P2 - 领养申请流程完善

**现状**：当前申请流程已可用，但存在以下可优化点：
- 同一宠物是否允许多人同时申请？（竞争性申请问题）
- 管理员审核的详细逻辑（锁定机制）

**两种方案对比**：

| 方案 | 设计 | 优点 | 缺点 | 推荐 |
|------|------|------|------|------|
| **方案A：锁定式申请** | 第一个申请人提交后，宠物自动进入"审核中"，其他人无法申请 | 避免竞争，符合直觉 | 效率低，一个申请被拒后宠物才开放 | 适合高端宠物 |
| **方案B：竞争性申请**（推荐） | 多人都可申请同一宠物，管理员择优审核 | 体现平台活跃度，学术上更有研究价值 | 需要管理员仔细审核 | **更适合毕业设计** |

**方案B详细设计（推荐）**：

```
用户提交申请 → 申请状态=待审核 → 其他用户仍可申请（状态不变）
管理员审核通过 → 申请状态=已通过 → 其他申请自动变为"已过期"
管理员审核拒绝 → 申请状态=已拒绝 → 其他申请继续等待审核
```

**功能完善清单**（1天）：

| 功能 | 说明 |
|------|------|
| 申请须知弹窗 | 用户申请前显示领养条件（稳定住所、不离不弃等） |
| 申请前查询 | 查询该宠物已有几个申请（"已有3人申请"） |
| 申请历史 | 用户可查看自己的所有申请记录及状态 |
| 管理员批量审核 | 一次审核多个人对同一宠物的申请 |
| 审核超时提醒 | 超过7天未审核的申请标红提示 |

### 4.7 P2 - 位置获取方案优化

**现状**：`shelter-recommendation.html` 已实现 Geolocation API 获取 GPS 位置。

**推荐方案：双模式自动切换（0.5天）**

```javascript
// 1. 优先尝试 IP 定位（桌面端友好，用户无感知）
async function getLocationByIP() {
    try {
        const res = await fetch('http://ip-api.com/json/?fields=status,country,city,lat,lon');
        const data = await res.json();
        if (data.status === 'success') {
            return { lat: data.lat, lon: data.lon, source: 'ip' };
        }
    } catch (e) {}
    return null;
}

// 2. GPS 作为补充（移动端）
async function getLocationByGPS() {
    return new Promise((resolve, reject) => {
        navigator.geolocation.getCurrentPosition(
            pos => resolve({ lat: pos.coords.latitude, lon: pos.coords.longitude, source: 'gps' }),
            err => reject(err),
            { timeout: 5000, enableHighAccuracy: true }
        );
    });
}

// 3. 智能切换逻辑
async function getUserLocation() {
    const ipLocation = await getLocationByIP();
    if (ipLocation) {
        getLocationByGPS().then(gps => {
            if (gps.accuracy < ipLocation.accuracy) {
                updateLocation(gps);
            }
        }).catch(() => {});
        return ipLocation;
    }
    return await getLocationByGPS();
}
```

**成本**：0元（IP-API.com 免费，45次/分钟限制足够演示）
**论文价值**：中——可补充"位置获取模块"设计说明，体现用户体验考量。

## 十五、v1.2.0 UI 深度优化计划（2026-04-11 新增）

> **版本**：V4.5 新增章节
> **目标**：让宠物救助网站整体风格更加温馨治愈、可爱，贴合宠物救助主题，提升答辩视觉冲击力。
> **总工时**：约 9 小时（2 天密集开发）

### 15.1 现状问题分析

| 你提到的问题 | 我的判断 |
|------------|---------|
| 颜色过于单调 | ✅ 准确。全靠 `--primary` 陶土珊瑚撑场，缺少色彩层次和点缀色 |
| 空白部分太多 | ✅ 准确。大片纯色背景（`#FDF6F0`）没有装饰性内容，显得"空" |
| 希望更温馨治愈可爱 | ✅ 宠物救助主题核心诉求，现版偏"干净冷淡"而非"温暖治愈" |

**额外观察**：
- 缺少宠物主题装饰元素（只用爪印背景，不够丰富）
- 卡片信息密度低，留白过多，视觉上"瘦"
- 缺少"氛围感"：没有插画、emoji组合、渐变图形营造温暖感
- 过渡动画不足，微交互比较简单
- 状态和分类的视觉区分不够
